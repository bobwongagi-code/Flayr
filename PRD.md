# Flayr · TikTok 带货短视频分析与提升工具

> 产品需求文档 v1.0 | 2026-05-22

---

## 1. 背景与目标

### 1.1 业务背景

公司签约了大量达人做 TikTok 带货短视频。达人整体水平参差不齐，缺乏对爆款视频的拆解能力，也不清楚自己的视频差在哪里。

现状痛点：
- 达人不知道爆款视频为什么爆
- 达人不知道自己的视频跟爆款差在哪
- 文字版建议达人看不懂也不愿意看
- 改进建议泛泛而谈，无法落地

### 1.2 产品目标

通过分析和拆解爆款短视频，对比达人视频，找出关键差距，并**生成一份可执行的改进方案**，让运营和达人能快速理解差距及改进方向。系统不生成替换视频或 AI 示意视频。

核心度量指标：达人视频的带货 GMV 提升。

### 1.3 目标用户

**直接用户**：公司运营团队（使用工具分析视频、生成提升方案）

**最终受益者**：签约达人（根据证据和执行建议改进下一条视频）

达人画像：
- 水平不高，不具备专业视频分析能力
- 不会主动阅读文字报告
- 需要直观、可视化的方式感知差距
- 能看懂视频但看不懂文档

## 2. 设计原则

以下四个原则贯穿所有功能设计，优先级从高到低：

| 原则 | 含义 | 反面示例 |
|------|------|---------|
| **可视化** | 通过关键帧、时间线和原视频证据呈现分析结果，文字只承担解释作用 | 输出一份没有画面证据的长文字报告 |
| **易于理解** | 标注用大白话，不用专业术语，达人一看就懂 | "需要提升视觉层次的信息密度" |
| **整体感** | 输出围绕完整视频链路组织的改进方案，不是散点清单 | 列出 10 个改进点让达人自己琢磨 |
| **突出重点** | 每次只聚焦 3~5 个关键提升点，追求最大 GMV 杠杆 | 面面俱到、事无巨细 |

### 2.1 提升建议的筛选标准

提升点可能有很多，但只保留 3~5 个，筛选依据：

1. **GMV 导向**：这个改动能不能让更多人买单？（不是"让视频更好看"）
2. **杠杆率**：投入的时间精力少，但对内容提升的撬动效果大
3. **具体可执行**：精确到"第 X 秒的 Y 元素怎么改"，不是宏大的方向性建议

反面示例：
- "画面要用 4K 分辨率" → 确实能提升画质，但对带货效果的杠杆太小
- "整体节奏需要加快" → 太泛，达人不知道具体哪里加快、加快到什么程度

正面示例：
- "开头第 1 秒把产品直接怼到镜头前，替换当前的自我介绍" → 具体、高杠杆
- "第 8 秒'而且它才 39 块'这句话之前，加一个 1 秒停顿" → 精确到秒、可直接执行

## 3. 功能架构

### 3.1 三种使用模式

```
模式 1：拆解（Breakdown）
  输入：一个爆款视频
  输出：结构化分析（爆款原因拆解）

模式 2：对比（Compare）
  输入：爆款视频 + 达人视频
  输出：差距诊断（逐维度差距对比）

模式 3：提升（Improve）
  输入：爆款视频 + 达人视频（或基于已有分析）
  输出：结构化改进建议 + 分析报告 HTML
```

三个模式可以串联使用（拆解 → 对比 → 提升），也可以单独使用。完整流程是模式 2+3 的组合。

### 3.2 完整工作流

```
Step 1 · 视频输入与解析
Step 2 · 视频拆解分析
Step 3 · 对比诊断
Step 4 · 筛选 Top 3~5 提升点
Step 5 · 生成 Top 3~5 条可执行改进建议
Step 6 · 输出交互式分析报告（HTML）
```

## 4. 详细功能设计

### Step 1 · 视频输入与解析

**输入**：用户提供视频文件路径（.mp4 等常见格式）

**处理流程**：

```
视频文件（.mp4）
  │
  ├── ffmpeg 提取关键帧
  │     → 全片逐秒截图（fps=1），输出为 frames/frame_0001.jpg, frame_0002.jpg ...
  │     → Hook 前 5 秒和 CTA 最后 5 秒加密截图（fps=2），输出到 focus_frames/
  │     → 用于后续大模型画面分析和人工复查
  │
  ├── ffmpeg 提取音频
  │     → 输出 audio.wav
  │     → 用于本地音频质检和阶段证据对齐
  │
  └── whisper 语音转文字
        → 默认自动检测口播语言，适配马来语、泰语、印尼语、英语等东南亚本地语言
        → 输出本地语言逐句文案 transcript.txt
        → 同时输出电商语境中文翻译 transcript.zh.txt，方便中国运营团队阅读
        → 格式：[00:01 - 00:03] "这款面膜真的太好用了"
        → 用于后续文案分析和改写
```

**输出**：
- 逐秒关键帧图片集
- Hook/CTA 加密关键帧图片集
- 原始音频文件
- 本地语言完整文案文本
- 电商语境中文翻译文案文本（重点处理黄色购物车、下方链接、价格优惠等带货话术）

**有效口播判断约束**：
- 不得用英文式空格分词或 word count 判断是否存在有效口播
- 泰语、中文、日语、高棉语、老挝语、缅甸语等语言可能没有空格，Whisper 输出可能是一长串文本
- 判断口播是否有效时，优先使用 `transcript.strip()`、Unicode 字符数、Whisper segment 数量、音频非静音段等信号
- 只有在转写为空、segment 为空且音频无有效人声时，才进入"无口播/画面推断文案"分支

**依赖**：ffmpeg、whisper（skill 首次运行时检测，未安装则引导安装）

### Step 2 · 视频拆解分析

**分析框架**：基于「最佳带货模板」（待提供的 markdown 文件）构建具体分析维度。

**初步框架**（将根据模板文件调整）：

分析沿"注意力漏斗"展开，每个阶段回答一个核心问题：

| 阶段 | 时间段 | 核心问题 | 分析要素 |
|------|--------|---------|---------|
| **钩子 Hook** | 0~3s | 用户凭什么停下来？ | 视觉冲击力、开场话术、悬念/冲突设置、声音吸引力 |
| **痛点激活 Pain** | 3~8s | 用户凭什么觉得跟自己有关？ | 场景共鸣、问题可视化、情绪触发方式 |
| **产品解法 Solution** | 8~20s | 产品怎么解决问题？ | 卖点展示方式、效果可视化、功能演示清晰度 |
| **信任转化 Trust** | 最后 3~5s | 用户凭什么现在就买？ | 社交证明、紧迫感/稀缺性、CTA 话术 |

贯穿维度：**整体制作力**（节奏/剪辑、画面构图、声音设计、字幕排版）

**分析拆分为两大维度**：

**维度一：产品卖点传递（基础层）**
- 核心卖点是否清晰
- 痛点是否精准触达
- 使用场景是否具象
- 价格锚点是否有效
- 信任要素是否充分（评价/销量/权威背书）

**维度二：视频表现力（放大器层）**
- 画面：产品展示角度、特写运用、场景切换
- 声音：语速、语气、情绪饱满度、BGM 配合
- 节奏：信息密度、停顿节奏、高潮设置
- 字幕：关键信息强调、价格/卖点的视觉突出

> 卖点传递是内容的底层价值，表现力是将这个价值最大化传递给观众的放大器。二者缺一不可。

---

### Step 3 · 对比诊断

**输入**：爆款视频分析结果 + 达人视频分析结果

**处理**：逐阶段、逐维度对比，识别差距

**输出**：

```
对比结果结构：
├── 阶段级对比（Hook / Pain / Solution / Trust）
│     每个阶段：
│     ├── 爆款做法摘要
│     ├── 达人做法摘要
│     ├── 差距描述（具体到秒数和元素）
│     └── 差距等级（大/中/小）
│
└── 维度级对比（卖点传递 / 表现力）
      ├── 哪些卖点没传递到位
      ├── 哪些表现力手法缺失
      └── 整体评分（直观对比用）
```

---

### Step 4 · 筛选 Top 3~5 提升点

**输入**：Step 3 的对比结果

**筛选逻辑**：

```
所有差距点
  │
  ├── 过滤：GMV 导向
  │     排除对带货转化无直接影响的差距
  │     （如纯美学问题、不影响决策的细节）
  │
  ├── 排序：杠杆率 = 预期效果 ÷ 执行难度
  │     执行难度考虑：达人能力、改动成本、是否需要重拍
  │
  └── 截取：取 Top 3~5
        正常视频取 3 个
        质量极差的视频可放宽到 5 个
```

**每个提升点的结构**：

```
提升点 #N：[一句话标题，大白话]
├── 问题：第 X 秒到第 Y 秒，[具体描述当前问题]
├── 对标：爆款视频在同样位置是怎么做的
├── 建议：改成 [具体的改法，精确到话术/画面/节奏]
└── 为什么：这个改动如何帮助提升 GMV
```

---

### Step 5 · 改进建议与素材参考

系统不生成达人音色视频、AI 示意视频或替换后的完整视频。改进点直接落在原视频的时间段、证据帧、执行动作和建议话术上；没有合适真实素材时明确要求补拍或补充素材。

---

### Step 6 · 交互式分析报告（HTML）

作为视频的补充，提供一份可视化的 HTML 报告给运营团队内部使用。

> 注意：这份报告主要给运营团队使用，也可以作为达人执行下一条视频的依据。

**报告结构**：

```
单文件 HTML，可在浏览器中直接打开

├── 顶部：视频信息卡
│     视频封面、时长、核心数据（播放/点赞/转化）
│
├── 漏斗拆解区
│     4 个阶段卡片（Hook → Pain → Solution → Trust）
│     每个卡片：时间段标记、关键帧截图、文案摘录、分析评语
│
├── 对比区（对比模式时显示）
│     左右分栏：爆款 vs 达人
│     差距用颜色标注（绿色=达标、红色=需提升）
│     评分雷达图或条形图
│
├── Top 3~5 提升点
│     每个提升点一张卡片
│     包含：问题描述 → 爆款参考 → 具体建议 → GMV 影响
│
└── 技术附录
      完整的逐帧分析时间线（可折叠）
      原始转录文案全文
```

**交互功能**：
- 点击时间戳可跳转到对应帧截图
- 对比区可左右滑动查看
- 提升点卡片可展开/折叠详情

## 5. 技术架构

### 5.1 Skill 文件结构

```
Flayr/
├── SKILL.md                        # Claude Code skill 主指令文件
├── README.md                       # 项目说明
├── references/
│   ├── analysis-framework.md       # 分析框架（基于最佳带货模板构建）
│   ├── hook-patterns.md            # 爆款钩子模式库（常见有效钩子分类）
│   ├── comparison-rubric.md        # 对比评分标准（每个维度的评分依据）
└── assets/
    └── report.html                 # HTML 分析报告模板
```

### 5.2 外部依赖

| 工具 | 用途 | 安装方式 |
|------|------|---------|
| ffmpeg | 视频帧提取、音频提取和证据片段处理 | `brew install ffmpeg` |
| whisper | 自动检测口播语言并转文字，获取带时间戳的文案 | `pip install openai-whisper` 或 `brew install whisper-cpp` |

Skill 首次运行时检测依赖是否就绪，缺少则给出安装指引。

### 5.3 数据流

```
输入                    处理                         输出
──────                 ──────                       ──────

爆款视频 .mp4  ─┐
                ├→  Step 1 视频解析  ─→  帧 + 音频 + 文案
达人视频 .mp4  ─┘
                        │
                        ▼
                  Step 2 拆解分析  ─→  结构化分析数据
                        │
                        ▼
                  Step 3 对比诊断  ─→  逐维度差距数据
                        │
                        ▼
                  Step 4 筛选提升点 ─→  Top 3~5 提升点
                        │
                         ▼
                  Step 6 生成报告
                         │
                         ▼
                      report.html
```

## 6. 输入输出规格

### 6.1 输入

| 字段 | 必选 | 说明 |
|------|------|------|
| 达人视频文件 | 是 | .mp4 等常见视频格式，本地文件路径 |
| 爆款视频文件 | 拆解/对比模式必选 | 同上 |

| 最佳带货模板 | 否 | markdown 文件，作为分析框架的补充参考 |

### 6.2 输出

| 输出物 | 格式 | 目标用户 | 说明 |
|--------|------|---------|------|
| 分析报告 | .html | 运营团队 | 交互式报告，可在浏览器打开 |
| 原始分析数据 | .json | 程序化使用 | 结构化的分析结果，便于后续系统集成 |
