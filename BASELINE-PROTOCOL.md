# Baseline 测量协议（常法，2026-06-21 起草）

> 起因：2026-06-21 发现——乱的根**不是「旧数据脏」，是「一直用单跑测量」**。
> temp=0 仍不可复现，执行分 B/C 在重跑间 ±1 档横跳，derive 阈值把它放大成 severity 翻档。
> 每个单跑命中率都是从会抖的分布里抽的一个点，没有误差棒。
> ground-truth-labels.md 早记过「单跑虚高~2分、稳定取众数」，本协议把它升为强制常法。
>
> **本文件是常法：以后每个 baseline / 实验都按它来，不按它得的数字一律不算数。**

## 三条铁律

### 铁律 1：单跑作废，一切数字 = N 次众数（稳定口径）+ 报一致度
- 任何命中率 / severity / B / C，**必须是 N 次重跑取众数**，N 见下（`[待 5× 定]`）。
- **同时报一致度**（N 次里几次落同档），把 wobble 摆明面、不藏。
- **一致度门槛（已钉死，不等 5×）**：`< 4/5`（或 `< 3/3`）同档 = **标红「不可信」**。即 N=5 需 ≥4 次同档、N=3 需 3 次全同，才算稳定结论；否则该格判「不可信」，不得用于汇报或拍板。
- 5× 实验本身也用这把尺读结果。

### 铁律 2：基准不可变 + 锁代码版本
- 基准 = 某个 git tag 下、固定样本集各 N 次众数。
- **存进只读目录、永不就地覆盖**。基准永远可追溯到「哪版代码 tag + 哪次 N 跑众数」。
- 改了代码要新基准 → 新 tag + 新目录，旧基准留着对照，不覆盖。

### 铁律 3：输入 / 基准 / 实验 三层物理隔离
```
runs/sample-X/                              ← 输入层，永不动
    analysis.json        （清单 videos + 输入 product；嵌的旧 stage_analysis 是死字节，每跑覆盖）
    _preprocess.json / benchmark/ / creator/ （帧/音/转写——重建贵，保住）
runs_baseline_v3/X/run_1..N/ + mode.json    ← 基准，锁 tag，只读，永不覆盖
runs_exp/<实验名>/X/run_1..N/ + diff_vs_baseline.json  ← 实验，每个独立版本，绝不碰基准
```
**实验 = (锁定代码 tag) × (N 次众数) × (对基准 diff)**。三个都钉住，不可能再「拿错误数据做实验」。

## 稳定口径计算器（把纪律写进代码，不靠记）
小脚本 `scripts/stable_verdict.py`（待写）：
- 输入：某样本 N 次 result。
- 输出：每阶段 **众数 severity + 一致度 (k/N) + 众数 B/C**；一致度 `< 4/5`（或 `<3/3`）**自动标红「不可信」**。
- 任何报告/实验结论必须经它产出，禁止手抄单跑值。

## 两个参数（5× 已定，2026-06-21 封版）
- **N = 5**。依据：carslan-b0 5× 实测，中段 S1-S4 的 B 抖到 3/5（wobble 凶），需至少 5 次取众数；沿用项目 Repeat5 先例。
- **facts 重跑策略 = 全流程 N 跑**（Step-0+facts+对比，不可 shortcut）。依据：5× 实测 video_facts **每次都不同**（benchmark/creator information 文本 5/5 唯一、benchmark 单元数 7↔6），**阶段1 抽取本身不确定**——不能「抽一次 facts 只跑阶段2」，必须全流程 N 跑。

## ⚠️ 5× 铁的发现：mode 口径必要但不充分
carslan-b0 即便取 N=5 众数，**S1/S2/S3 一致度仍只 3/5 = 仍🔴**。
→ mode 口径能**诚实标出**「这些阶段不可信」（达成本协议目的：不再假装可信），但**盖不住抖动本身**。
→ **子②（阶段1 感知不确定性）是真缺陷，必须单独治**（见 TODO §0/执行分线子②）；光靠多跑取众数，wobbly 样本的中段永远🔴。
→ 含义：baseline-v3 出来后，**中段 wobbly 样本/阶段会带🔴标记**——这是诚实，不是 baseline 失败；它精确定位了子② 该治哪几格。

## Stage1 不稳的两种病（2026-06-23 诊断，carslan 实证，须分开治）

| | **覆盖漂移 Coverage Drift** | **措辞漂移 Lexical Drift** |
|---|---|---|
| 现象 | 重磅事件时有时无（S5 泼水测试 5 跑 2 有 3 无） | 同一事实每跑换词（S4 Before/After 5/5 措辞不同但语义全中） |
| 机制 | **注意力分配**随机：有限叙述预算里模型每次自由选「提哪些瞬间」，发生在输出采样层**上游** | **token 采样**随机：自由散文表达 |
| temp=0 能压？ | **压不住**（不在采样层） | 能大幅压制 |
| 伤下游？ | **伤**（事件缺失 → 阶段2 判断翻档），这是真根 | 不伤（语义稳，阶段2 照样拿到事实） |
| 唯一有效修法 | **清单化**（自由叙述 → 逐项 yes/no 扫描） | 不用治；只是别拿它当不稳指标 |

**关键**：此前全部手段（temperature / prompt 硬约束 / JSON 强制）针对的是措辞漂移和阶段2 判断漂移，**对覆盖漂移几乎无效**。清单化不是收紧 prompt，是任务范式的根本改变——让「泼水测试」从「靠模型想起来提」变成「被点名问的 yes/no」。

**根的物理位置（2026-06-23 代码级确认）**：覆盖漂移源于 Stage1 两个地基**到场但是软的**——`build_video_fact_payload` 里品命题只作软提示（`obs_hint`，payload.py:245，且 category 痛点根本没注入）、结构库只有粗 6 阶段 `functions` 标签（事件目录 S4-A~F/S3-A~E 没给）。软地基 = 模型没有「必须逐项核查的高价值事件硬清单」→ 泼水只是众多可选细节、被随机丢。**清单化的本质 = 把两个地基在 Stage1 从软提示打成硬地板；注入管道（product_foundation→obs_hint）已现成。**

## 总纲先于清单：品地基+结构库是设计层，清单是执行层

**这条比下面所有清单设计规则都靠前——清单的合法性来自总纲，不来自清单自己。** 清单是执行机制，不是设计权威。设计层的权威始终是两个地基：

- **品地基（品 / 品的特征 / 命题）= 为什么应该存在证据**
- **结构库 = 那个证据在内容里长什么样**
- **清单 = 把这两件事变成可靠执行的 yes/no 检测**

```
命题（为什么）          ── 品地基
  ↓ 由结构库翻译
结构事件（长什么样）    ── 结构库
  ↓ 由清单项稳定检测
yes / no（有没有）      ── 清单
```

**丢了总纲的后果**：通用清单只能查「有没有数字 / 有没有前后对比」（内容格式核查）；带总纲的清单查的是「这个数字 / 对比有没有在结构库定义的正确形态上验证这条命题」。同一条视频两个答案可能相反——格式有、命题验证缺 = 对投放无价值，通用清单却误判为「有效果演示」（has_effect_demo 全场 True 的同类病）。

**命题相关性焊进锚定、不在运行时问**（化解「纯观察 vs 验证命题」张力）：二者直接合一会出事——「这数字支不支撑命题」是判断，破观察墙、把抖动请回来。化解口：命题相关性**设计时焊进锚定，不在运行时让模型判断**。命题「持久」→ 结构库「极限测试 / 持妆计时」→ 锚定「泼水 / 持妆 X 小时」→ 模型只观察「有没有这个被锚定的形态」。问的不是「有没有数字」（会把价签算进），是「有没有持妆时长这类数字」。**锚定是命题的观察化身——省了它，命题相关性只能靠运行时判断找回、抖动原路返回。故锚定是载荷不是装饰、不可省、必须从品地基加载**（「什么算数字」在底妆和精华液里不同）。

**清单项必须带命题索引**：每项旁标注它在验证哪条命题；答不上「我在验证哪条命题」的项，要么不该在、要么设计错层级。命题变 → 清单项优先级跟着变（清单自带命题索引、被命题拉着，不漂成历史积累的审计条目）；结构库更新（新赛道新高价值结构事件）→ 评估增补检测项。**总纲是活的，清单跟着活。**

### Stage1 稳定度量（换掉文本相似度）
**Stage1 稳定性 = 清单项跨跑一致率**，不是文本去重/相似度。S5 泼水在旧指标是「5/5 视觉描述全不同」（表面、误导），新指标是 `has_water_resistance_test = T/F/T/F/F`（0.4 一致率 → 触发 kill）。同一现象，只有后者能触发正确响应。

### 清单设计原则（防自身引入漂移）
- **静态预设、按赛道锁**：清单上线前人工定，**绝不运行时动态生成**（动态生成 = 把漂移前移一步）。
- **从 Stage2 字段往上推，不从视频内容往下枚举**：每条清单项 = 某个 Stage2 评分字段的直接事实前体（一对一/多对一映射）。**穷举是问「视频里有什么」（无边界）；清单是问「Stage2 需要知道什么」（目标驱动、自然有界）**。
- **构念 + 锚定两层**：问法带例子——「有没有任何形式的产品性能压力测试？（如泼水/出汗/持妆计时）→ yes/no」。纯抽象重新引入歧义（模型自裁→判断噪声回来），带例子既泛化又锚定。构念跨赛道稳定，锚定示例域内可替换。
- **上界 + 下界都设**：上界 = Stage2 需要事实输入的评分字段数（超出 = 收集 Stage2 用不到的信息，只增成本不增价值）。**下界 = 每项必须有 ≥1 Stage2 字段直接消费**，无消费方不进。
- **两道合法性闸对齐**：① 每项必须验证 ≥1 命题（总纲）；② 每项必须有 ≥1 Stage2 消费（下界）。二者不一致时（命题有、Stage2 没消费）= **Stage2 缺口、该补 Stage2 维度，不是丢清单项**（Stage2 维度本就该从命题派生）。下界不得反过来阉掉总纲该管的命题。
- **赛道选择不进漂移面**：清单按赛道选，赛道从**品地基（Step-0 已建 category_profile）**拿——产品属性、上线定死，不每跑看视频猜。否则漂移从清单内容前移到清单选择。

### 三个风险 → 三个动作（不只命名）
- **臆造风险 → 问法「扫描后作答」两步**：① 先一句话描述本段所有相关事件；② 基于你刚才的描述逐项答 yes/no。模型答的依据是自己刚生成的描述、不是凭空确认——两步间形成自校验，比反臆造闸更结构性。
- **观察/判断墙 → 每项上线前过设计检验**：检验问题=「**给定锚定，外行只凭画面、不做质量判断，能不能答明确 yes/no？**」能 → 观察、可硬化；不能（要推断好不好 / 要领域判断算不算）→ 判断、退回 Stage2。**设计阶段就跑，不靠上线后一致率发现**（has_effect_demo 当初被当「显然是观察」放进去，结果是半判断、美妆全场 True）。
- **别灌全目录 → 上下界 + 只灌有界清单**：不灌整本结构库（臃肿、注意力摊薄、反更漏），只灌有界清单（= Stage2 字段数那几个构念）。

### 实施顺序（管道已通、路径比新建短，但「过检验」一步最易跳、不可跳）
清单项设计 → **每项过观察/判断检验** → 补品地基锚定示例（A~G 锚定全从品地基加载，非只痛点项）→ 改 prompt 为「扫描后作答」→ 补 Stage1 埋点 → 跑 5 次验一致率（先裸一致率，全量再上 κ/CI）。

### 埋点（坐实根，与 colorkey 补跑并行、不互等）
每跑存两份：①`video_facts_raw_*.json`（Stage1 完整自由文本，**已加入 run_batch**）；②`video_facts_checklist_*.json`（清单 yes/no 结构化，待清单设计完再加）。

### 探针方法论（2026-06-23 F 项实证）
- **观察/判断线极易滑——每个清单项都必须过检验，没有例外。** F 项 ① 第一版写成「口播有没有提到具名硬背书**来源**」，这个「来源/背书」让模型去判「随口说'有个 halal 标'算不算援引背书」→ 它做了判断、答了判断（① no），不是观察。纯观察版＝「口播有没有【出现】halal/KKM/认证 等**词**」（与「算不算有效背书」彻底分开，后者归阶段2）。**连精心设计的项都会偷渡判断，措辞里带评价词（来源/有效/算不算/够不够）就是信号。**
- **Arm A（两步：先列原句再对照）有超出 yes/no 的诊断价值。** B/C 只吐最终答案，判断污染时看不出错在哪；Arm A 的 Step1 强制展示「听到/看到了什么」，一眼定位是「没感知到」还是「感知到了但判断错」。F 项 ① 正是靠 Arm A 的 Step1 列出「KORANG ADA HALAL」才坐实：模型**听到了**、是**判断**把它否掉的。**B/C 答案可疑时，跑一次 Arm A 看推理即可定位问题在哪一步。后续每个项设计都适用。**
- **ground truth 也可能被判断污染。** F 项 ① 起初定 correct=yes（「口播提到背书」），其实那是判断后的答案；纯观察真值是「halal 词出现＝yes」。定标时也要分清「词在不在」（事实）vs「算不算」（判断）。
